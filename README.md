# Personal Telegram Reminder & People Bot

Личный Telegram-бот на aiogram 3.x: напоминания с гибкими алертами и
мини-CRM по знакомым с дедупликацией имён. Свободный текст разбирается через
Anthropic API (модель `claude-haiku-4-5-20251001`) в структурированный intent.

## Возможности

- ⏰ Напоминания из свободного текста («позвонить бабушке сегодня в 12»),
  несколько вариантов «напомнить заранее», отложенные и повторяющиеся
  напоминания (каждый день / каждую неделю).
- 👤 Мини-CRM: карточки людей с алиасами, заметками, днями рождения;
  дедупликация при создании нового человека учитывает русские уменьшительные
  формы (Валера/Валерчик/Валеныч/Валерий, Саша/Александр, Дима/Дмитрий, ...)
  и показывает до 3 похожих кандидатов кнопками вместо создания дубликата.
- 🎂 Авто-напоминания о днях рождения — ежегодное напоминание создаётся/
  обновляется/удаляется автоматически при любом изменении даты рождения.
  Время по умолчанию — `BIRTHDAY_ALERT_TIME`, но каждый пользователь может
  переопределить его через `/settings`.
- ⚙️ `/settings` (или фраза «настройки») — время уведомлений о днях
  рождения: пресеты (09:00/12:00/15:00/18:00/20:00) или своё время текстом
  («16:45», «16 45», «в 5 вечера»). При изменении ВСЕ существующие
  напоминания о днях рождения сразу переносятся на новое время (не после
  перезапуска). Кнопка «🔔 Тест уведомления» ставит реальное тестовое
  уведомление через 2 минуты через тот же планировщик — способ проверить,
  что уведомления действительно доходят.
- 🔎 Полнотекстовый поиск по заметкам (SQLite FTS5) — «кто любит рыбалку?».
- 🔒 Бот приватный — отвечает только владельцу (`ALLOWED_USER_IDS`).
- 💾 Напоминания переживают перезапуск — задачи APScheduler персистятся в
  отдельном SQLite-файле.
- 📦 `/export` — бэкап всех людей и напоминаний в JSON-файл, присылается
  документом в чат.
- 📊 `/stats` — количество людей, заметок и активных напоминаний в базе.
- 🗂 Все файлы на диске (обе БД, логи, временные экспорты) живут строго
  внутри `DATA_DIR` — важно для деплоя на Railway с примонтированным томом.
- 🧹 Дубли заметок: при сохранении заметка сверяется (fuzzy, порог 85) с уже
  существующими у этого человека — если похоже, всё равно сохраняет, но
  предупреждает; на карточке есть кнопка «Почистить дубли» со списком
  похожих пар и кнопками удаления.
- 🧼 Интерфейс не копится стрелками в чат: у карточки/списков есть
  «✖️ Закрыть» (удаляет сообщение), а после финального действия (создать,
  отменить, выбрать кандидата и т.п.) клавиатура у сообщения убирается.
- ❓ Вопросы о запланированном: «напомнишь ли ты мне это 12 июля?» —
  отвечает по существующим напоминаниям, не заводит новое.
- 📅 `/today` (или «что сегодня») — сегодняшние напоминания и дни рождения.
- 💾 Автоматический дневной бэкап в `DATA_DIR/backups` (последние 7),
  плюс ручной `/export`.
- 🛡 Надёжность: ретрай при сбое Anthropic API с честным сообщением
  пользователю; глобальный error-guard (бот не падает от исключений);
  запас на опоздание напоминаний 24ч с пометкой «(с опозданием)» и
  повторной отправкой при сбое Telegram; проверка целостности jobstore
  при старте; корректное завершение по SIGTERM (redeploy на Railway).
- 🎂 Если известен год рождения — в уведомлении будет «Исполняется N».

## Стек

Python 3.11+, aiogram 3.x, SQLAlchemy 2.x (async, SQLite/aiosqlite),
APScheduler (AsyncIOScheduler + SQLAlchemyJobStore), Anthropic API, rapidfuzz.

## Установка

1. **Создать бота у BotFather**
   - Напишите [@BotFather](https://t.me/BotFather) в Telegram, отправьте
     `/newbot`, следуйте инструкциям и сохраните выданный токен.
   - Узнайте свой Telegram user ID (например, через [@userinfobot](https://t.me/userinfobot)) —
     он понадобится для `ALLOWED_USER_IDS`, чтобы бот отвечал только вам.

2. **Установить зависимости**

   ```bash
   python3 -m venv .venv
   source .venv/bin/activate
   pip install -r requirements.txt
   ```

3. **Настроить `.env`**

   ```bash
   cp .env.example .env
   ```

   Заполните файл:

   ```env
   TELEGRAM_BOT_TOKEN=токен-от-BotFather
   ANTHROPIC_API_KEY=ваш-ключ-Anthropic
   TIMEZONE=Europe/Kyiv
   ALLOWED_USER_IDS=ваш_telegram_id
   DATA_DIR=./data
   BIRTHDAY_ALERT_TIME=15:00
   ```

   `DATA_DIR` — папка для всех файлов, которые должны пережить перезапуск
   (обе БД, логи, временные экспорты для `/export`). **На Railway** (или
   любой платформе с эфемерной файловой системой) обязательно укажите тут
   путь к примонтированному volume, например `/app/data` — иначе база
   исчезнет при каждом редеплое.

4. **Запустить бота**

   ```bash
   python3 bot.py
   ```

   При первом запуске автоматически создастся `<DATA_DIR>/bot.db` (основная
   база) и `<DATA_DIR>/scheduler.db` (хранилище задач APScheduler, отдельный
   файл — так избегаются конфликты блокировок SQLite между async-движком и
   синхронным jobstore'ом). Логи пишутся в `<DATA_DIR>/bot.log` и в консоль.
   При старте в лог выводятся резолвленные абсолютные пути обеих БД —
   удобно проверить в логах Railway, что они указывают на смонтированный том.

## Структура проекта

```

config.py               — конфигурация из .env
db/
  models.py             — модели SQLAlchemy (Person, Reminder, UserSettings, ...)
  repo.py               — доступ к БД, инициализация FTS5
handlers/
  common.py             — /start, /help, диспетчер intent → обработчик, access control,
                          общий обработчик «✖️ Закрыть»
  reminders.py          — сценарии напоминаний
  people.py             — сценарии мини-CRM (люди, дедупликация, поиск, заметки)
  settings.py           — /settings: время др-уведомлений, тестовое уведомление
  states.py             — FSM-состояния диалогов
  ui.py                 — общие UI-хелперы: кнопка «Закрыть», пустая клавиатура
services/
  llm_parser.py         — обращение к Anthropic API, разбор JSON-интентов,
                          парсинг голого времени (parse_time_of_day) для /settings
  conversation.py       — короткая история диалога для контекста классификатора
                          + детерминированная проверка явных слов отмены
  scheduler.py          — APScheduler: планирование и срабатывание напоминаний
  person_matcher.py     — fuzzy-поиск дублей людей (rapidfuzz + словарь
                          русских уменьшительных имён + эвристика общего префикса)
  note_matcher.py       — fuzzy-поиск дублей ЗАМЕТОК у одного человека
  birthdays.py          — авто-напоминания о днях рождения (+ idempotent sync при
                          старте, per-user время из /settings, reschedule-all,
                          тестовое уведомление)
scripts/
  verify_birthdays.py   — самопроверка: DATA_DIR, создание/обновление/удаление
                          дня рождения, идемпотентная синхронизация при старте
  verify_dedup_flow.py  — самопроверка сценария дедупликации: кандидаты,
                          алиас по подтверждению, ветка без совпадений
  verify_ui_and_dedup.py — самопроверка дублей заметок, «Почистить дубли»,
                          маркера «(+N заметки)», кнопки «Закрыть» и
                          зачистки клавиатуры после финальных действий
  verify_settings_flow.py — самопроверка /settings: меню, пресеты, своё время
                          (реальный вызов классификатора), немедленный перенос
                          существующих напоминаний, тестовое уведомление,
                          /export включает settings
  eval_classifier.py    — прогон ~35 фраз через реальный Anthropic API,
                          таблица pass/fail для классификатора intent'ов
                          (вызывает llm_parser.parse_message НАПРЯМУЮ)
  eval_dispatch.py      — тот же принцип, но через РЕАЛЬНУЮ точку входа
                          (handlers.common.dispatch_text) — ловит баги вида
                          «сообщение вообще не дошло до LLM»
  eval_dedup.py         — ~14 пар уменьшительных имён, таблица pass/fail
                          для fuzzy-дедупликации (без обращения к API)
```

## Классификатор intent'ов: контекст и детерминизм

- Классификатор вызывается с `temperature=0` для воспроизводимости.
- Каждому вызову передаётся текст последних сообщений (`services/conversation.py`,
  до 6 последних реплик пользователя/бота), чтобы короткие ответы вроде
  «сегодня», «в 12», «да» правильно понимались как ответ на предыдущий
  уточняющий вопрос бота, а не классифицировались из ничего.
- Отмена/удаление (`delete_reminder`) распознаётся ДЕТЕРМИНИРОВАННО по явным
  словам («отмени», «удали», «убери», «не надо», «стоп») ещё ДО обращения к
  LLM — так фразы вроде «поздравить с днём рождения» никогда не могут быть
  ошибочно приняты за отмену, независимо от модели.
- УТВЕРЖДЕНИЕ vs ВОПРОС про день рождения не путается из-за слова «У» в
  начале: «У Валерчика др 12 июля» — это факт (`add_person_info`), а «Когда
  др у Валерчика?» / «У кого др в июле?» — вопрос (`query_person`).
  Ключевой сигнал для модели — наличие конкретной даты в сообщении.
- Каждый вызов классификатора логируется одной строкой
  (`LLM_CLASSIFY | in=... | context=... | out=...`) — удобно грепать в логах,
  если бот на что-то отреагировал не так.
- `python3 scripts/eval_classifier.py` прогоняет набор тестовых фраз (включая
  все сообщённые баги) через реальный API и печатает таблицу pass/fail —
  использовать при любых правках промпта.

### ⚠️ Важно: eval_classifier.py тестирует ТОЛЬКО классификатор в изоляции

`eval_classifier.py` вызывает `llm_parser.parse_message()` напрямую — он не
проверяет, что сообщение вообще ДОШЛО до классификатора. Реальный баг («У
Валерчика др 12 июля» обрабатывалось за 26-109 мс с нулём строк
`LLM_CLASSIFY` в логах) был именно в этом: `dispatch_text` содержал
pre-LLM shortcut на regex/подстроку, который перехватывал сообщение ДО
вызова LLM — `eval_classifier.py` в принципе не мог это поймать, потому что
не проходит через `dispatch_text`.

`python3 scripts/eval_dispatch.py` — эта проверка вызывает
`handlers.common.dispatch_text()`, то есть реальную точку входа aiogram, и
проверяет: (1) итоговый ответ бота, (2) был ли реально вызван классификатор
(через monkeypatch-счётчик поверх `llm_parser.parse_message`), (3) не был ли
ответ подозрительно быстрым (<150 мс) для сообщения, которое должно было
уйти в LLM (реальный вызов Anthropic API занимает 300+ мс). Использовать
этот скрипт, а не только `eval_classifier.py`, для проверки на регресс.

### Карта pre-LLM route'ов (всё, что может обработать сообщение до LLM)

| Что перехватывает | Где | Статус |
|---|---|---|
| Команды (`/start`, `/help`, `/cancel`, `/stats`, `/export`, `/people`, `/reminders`) | `common.py`, `people.py`, `reminders.py` | ОК — явные команды, не текст на естественном языке |
| Точное совпадение `"мои люди"` | `people.py: people_cmd` | ОК — точное совпадение всей строки (не подстрока), не может случайно сработать на несвязанном сообщении |
| Точное совпадение `"мои напоминания"` / `"напоминания"` | `reminders.py: list_reminders_cmd` | ОК — то же самое, точное совпадение |
| FSM-состояния ожидания конкретного ответа (`waiting_time`, `waiting_custom_offset`, `waiting_person_name`, `waiting_name_edit`, `waiting_new_note`, `waiting_rename`) | `reminders.py`, `people.py` | ОК — это не эвристика по ключевым словам, а осознанно ограниченный контекст: бот только что задал конкретный вопрос и ждёt ответ именно на него (некоторые из них всё равно вызывают классификатор, например `waiting_time`) |
| Явные слова отмены (`отмени`, `удали`, `убери`, `не надо`, `отмена`, `стоп`) | `services/conversation.py: is_explicit_cancel`, вызывается в `dispatch_text` и во всех FSM-обработчиках | ОК, сознательно — единственный оправданный pre-LLM shortcut: безопасность (отмена не должна зависеть от вероятностной модели) |
| ~~Regex по «др»/«рождени» + подстрока месяца~~ | ~~`people.py: detect_birthday_month_query`~~ | **УДАЛЕНО** — именно это ловило «У Валерчика др 12 июля» и другие сообщения с датой рождения, отправляя их в список дней рождения за месяц мимо классификатора. Теперь ВСЕ вопросы/утверждения про дни рождения (включая «у кого др в июле?») идут через LLM, который возвращает `birthday_query_month` для групповых запросов |

## Дедупликация людей: уменьшительные имена

- При СОЗДАНИИ нового человека (`add_person`) порог fuzzy-совпадения ниже
  (60 вместо строгих 80) и подключён словарь русских имён-семейств
  (Валера/Валерчик/Валеныч/Валерий, Саша/Александр, Дима/Дмитрий, ...) плюс
  эвристика общего префикса (≥4 букв) — так фразы вида «новый знакомый
  Валеныч» находят уже созданного «Валерчика», а не тихо создают дубликат.
- Показывается до 3 кандидатов кнопками с 1-2 известными фактами:
  «Нашёл похожих. Это кто-то из них?» + «➕ Нет, создать нового» + «❌ Отмена».
- При выборе кандидата бот спрашивает, сохранить ли упомянутое имя как алиас
  («Добавить «Валеныч» как алиас к Валерчику?»); факты/заметки из сообщения
  добавляются человеку в любом случае — да/нет влияет только на алиас.
- Добавление ФАКТОВ к уже существующему человеку (`add_person_info`)
  осталось на строгом пороге (80) с прежним поведением — ложные совпадения
  там дороже (можно молча приписать заметку не тому человеку).
- `python3 scripts/eval_dedup.py` — таблица pass/fail по парам уменьшительных
  имён, без обращения к API.

## Заметки по реализации

- FTS5 не умеет в русскую морфологию «из коробки» (не найдёт «рыбалка» по
  заметке «любит рыбалку», так как это разные словоформы) — поиск по
  заметкам поэтому в первую очередь пробует FTS5-запрос, а если он ничего
  не находит, откатывается на `LIKE`-подстроку.
- Бот не падает от неожиданного/невалидного ответа модели: JSON парсится
  через `json.loads` с обработкой ошибок, при сбое возвращается intent
  `unknown` и бот вежливо переспрашивает.
